# Harden & Provenance — trustworthy runs, replayable by record (STAGE 0-1)

This reference is the depth behind **STAGE 0 — Harden** and **STAGE 1 — Provenance** of the [../SKILL.md](../SKILL.md) flight plan. It governs the half of the run that happens *before any formal number is produced*: turning the MVP that survived [crucible](../../crucible/SKILL.md) into experiment code you can trust, and stamping every run so it can be replayed by record. Hardening makes the number *correct*; provenance makes it *replayable*. Neither is optional, and both are done once, up front, under the frozen interface — because everything downstream (the idempotent pipeline, the anti-mixing integrity rule, one-command regeneration) assumes a run is both trustworthy and replayable, and inherits the rot if it is not. It backs the five gated checks across the two stages: harden's `hardened-to-research-standard` and `regression-against-mvp`, and provenance's `randomness-recorded-not-eliminated`, `run-archive-minimal-sufficient`, and `clean-commit-gated`. The fact every move below is judged against:

> Harden to the research standard and no further — trustworthy and reproducible, not maintainable for a decade; and manage randomness by *recording* it, not killing it, so the same (config, seed) is a reproducible unit — replayable to the same result, bit-for-bit where the determinism switches are on — and the evidence chain closes itself.

---

## Why hardening and provenance are one document

They are separated into two stages because they gate separately, but they answer one question together: *can someone — a reviewer, a co-author, you in three months — reconstruct exactly what produced this number and trust that it is right?* Hardening answers the *trust* half: the number is what the protocol says it is, not an artifact of per-method metric drift or a silent infeasible solution. Provenance answers the *reconstruct* half: the run carries its own commit, config, environment, data version, and seed, so it can be replayed against any later question. A run that is hardened but not provenanced gives a correct number you cannot defend later; a run that is provenanced but not hardened lets you replay a wrong number perfectly. You need both, and you build them in this order — harden first (the code that runs must be right), then provenance (record the run that the right code produces) — because there is no point stamping the provenance of a run whose numbers you have not yet made trustworthy.

The lever is the suite's: the hardening refactor is *exactly* the dull, well-bounded labor the agent exists for — config-izing every magic number, collapsing four copies of metric code into one path, writing the feasibility checker, wiring the run archive. That is hours of mechanical refactoring handed to an overnight operator. But its gradient does not point at *correct*; it points at *clean and passing*. Left soft it will "improve" the code while subtly shifting a number, fold the feasibility check into the solver it is supposed to police, or declare the migration done because the new tests pass — and a refactor that reads clean and passes its own tests is *not* evidence the numbers held. **The agent is the means — here, the refactorer working under your frozen interface — never the author of the numbers.** Command it to harden and to record; verify the one thing only the regression match can show: that the numbers did not move.

## STAGE 0 — Harden to the research standard (`hardened-to-research-standard`)

The MVP got you to a hypothesis. Experiment code has a different bar: every number it produces must be *trustworthy* — defensible to a reviewer, reproducible by a stranger. That bar is reached by exactly **four investments**, and crucially by *not* over-investing past them. The check `hardened-to-research-standard` is clear when all four are in place under a frozen interface.

### The four investments that earn their keep

**(1) Config-ize everything — zero magic numbers in code.** Every parameter that defines a run — instance size, time limit, learning rate, the seed, the metric's tolerance, the hardware tier — lives in **one config file**, and that file defines the run *completely*: hand someone the config and they know exactly what was run, with nothing hidden in a literal buried on line 340 of a solver. The test is mechanical: there is no number in the code that changes the experiment's meaning. A magic number in code is a parameter you cannot vary, cannot record in the archive, and cannot disclose in the paper — and it is the seam where two "identical" runs silently differ.

**(2) Collapse metric computation to a single shared code path.** Every method's results flow through the *same* metric code — one function computes the objective, the gap, the runtime, for your method and for every baseline alike. This is the single highest-value hardening move, because **per-method metric code is the number-one source of misaligned experiments.** When each method computes its own score, two implementations of "the same" metric drift — one counts setup time, one does not; one rounds, one truncates; one measures the gap against a different reference — and the comparison silently stops being apples-to-apples. The blowout in your table is then a metric artifact, not a result, and it is exactly the kind of difference a reviewer reproduces and names. One metric path makes the comparison *structurally* fair: there is no second implementation to drift.

**(3) An independent feasibility checker that shares no logic with any solver.** Every solution any method produces is re-checked, on *every* run, by a verifier that is written **independently of all solvers** — it imports none of their code, reuses none of their constraint logic, and would catch a violation the solver's own internals are blind to. This is the strongest single trap for the agent's silent infeasible-solution bug. A research number that "soars" — suddenly far better than anything plausible — is, nine times in ten, an *infeasible* solution: the solver found a "great" answer by quietly violating a constraint, and its own code, sharing the buggy assumption, blessed it. A checker that shares the solver's logic shares the solver's blind spot and waves the same bad solution through. Only an *independent* checker, run on every produced solution, catches it. (This is why STAGE 3's operator boundary treats a feasibility-check failure as a code failure that is *never* auto-retried — the checker firing is signal, not flakiness.)

**(4) Freeze the interface.** Once confirmation starts, the method's **external behavior does not change.** The inputs it accepts, the outputs it produces, the meaning of its parameters — frozen. To change any of them is not an edit; it is **cutting a new version number.** This is what makes every later number comparable: a table that mixes results from two different external behaviors of "the method" is comparing two different methods under one name. Freezing the interface is also what makes the hardening refactor *safe* to hand the agent — it may restructure the internals freely (that is good work to delegate), but the frozen external contract is the wall it cannot move, and any change to it surfaces as a version bump you must approve.

### Why over-abstraction is wasted

The standard is **trustworthy and reproducible, not maintainable for a decade.** Experiment code is not a product; it is the apparatus that produces a paper's evidence and then is archived. Design-pattern polish, deep inheritance hierarchies, a plugin system for solvers you will never add — these are real engineering virtues that are *wrong here*, because they cost time the run cannot spare and buy a longevity the run does not need. Invest in the four things that make the numbers trustworthy and reproducible; spend nothing on maintainability past the life of the experiment. The hardening refactor itself — config-izing, unifying the metric path, writing the checker — *is* good, substantial work to hand the agent; just hand it **under the frozen interface**, so the restructuring cannot drift the external behavior the numbers depend on.

### Regression against the MVP (`regression-against-mvp`)

Here is the trap the whole stage exists for. You have refactored the MVP into clean experiment code; the new code reads well and passes its own tests. **None of that is evidence the numbers held.** A clean refactor that passes its tests can *silently change behavior* — a reordered accumulation, a changed default, a subtly different boundary condition — and the tests, written against the new code, pass anyway because they encode the new (wrong) behavior. The only evidence that the migration preserved the numbers is a **direct regression match against the old MVP.**

Before migration counts as done: run the **old MVP** and the **new code** on **several instances**, with the **same config and the same seed**, and the numbers must **match.** Not "look similar," not "pass the new test suite" — *match*, to the precision the metric is reported at. A mismatch is a behavior change the refactor introduced, and you find its source before you proceed; a match is the one thing that licenses you to throw the MVP away and run formally on the new code. This is the regression-against-the-MVP discipline, and it is non-negotiable because it is the only check that distinguishes "I rewrote the code" from "I rewrote the code *and it still computes the same thing*." The agent will report the refactor done when its tests pass; you require the regression match.

## STAGE 1 — Provenance (make every run replayable by record)

A formal run that cannot be replayed is not evidence — it is an anecdote. Provenance is the discipline that makes every run reconstructable from its own record: the randomness it used, the exact code and environment it ran under, and a single id that ties them together. The check set across this stage is clear when randomness is *recorded* (not eliminated), the run archive is *minimal-sufficient*, and the run *refuses to start* from a dirty tree.

### Manage randomness by recording it, not killing it (`randomness-recorded-not-eliminated`)

The reflex is to make experiments deterministic by setting one global seed everywhere. That is the wrong move, and it quietly destroys comparability. The right move is to manage randomness **by recording it.**

Each independent random source gets its **own independently derived seed, separately recorded.** Instance generation, the initial solution, the algorithm's internal randomness, data shuffling — these are *different* streams, and each is seeded from its own derived seed, logged on its own. The reason is mechanical and decisive: with one global seed feeding everything in sequence, **editing one piece of code shifts every downstream stream.** Add a line that draws a random number during initialization and the instance generator, the shuffler, and the solver's internals all receive different values than before — and your new run is no longer comparable to the runs you already recorded, through code you did not even touch. Independently derived seeds wall the streams off from each other: changing the initialization code moves the initialization stream and *only* it, and every prior run on the other streams stays exactly comparable. You are not trying to make randomness go away; you are trying to make it **reproducible and stable under edits**, and recording each source independently is how.

**GPU determinism is its own problem.** Even with seeds fixed, GPU computation is not automatically reproducible: cuDNN selects **non-deterministic ops** (some convolution and reduction kernels race), and **float accumulation order** varies with parallel scheduling, so the same inputs can yield bit-different outputs run to run. For the **key comparison experiments**, turn the **determinism switches on** — force deterministic algorithms, fix the accumulation — and **accept the speed cost.** That cost is real and you do not pay it everywhere; you pay it where the number must be replayable and a reviewer might rerun it. A comparison whose result flips on float-accumulation noise is not a result.

The target of all of this is **task-level determinism**: the same `(config, seed)` pair yields the **same result on any rerun — bit-for-bit under a pinned environment with the determinism switches on, and reproducible across machines up to the metric's reported precision** (which is why the archive records the hardware model and the lockfile). This is not a nicety — it is the *precondition* for two later mechanisms. It is what makes STAGE 2's **idempotent resume** correct (the executor can skip a cell because rerunning it would produce the identical number, so a completed cell is genuinely done). And it is what makes STAGE 4's **anti-mixing integrity** possible (you can prove two numbers belong in one table because they were produced under reproducible, identical conditions). Without task-level determinism, "resume" is a guess and "these numbers are comparable" is a hope.

### The minimal-sufficient run archive (`run-archive-minimal-sufficient`)

Every formal run **auto-records**, under one **run id**, the minimal set of facts sufficient to reconstruct it:

- **git commit hash** — the exact code that ran.
- **full config snapshot** — every parameter of the run (this is why config-izing in STAGE 0 comes first: the snapshot is only complete if nothing is a magic number in code).
- **environment lockfile** (or **container image tag**) — the exact dependency versions, or the image they were frozen in.
- **data version** — the **file hash** of the input data, so a silently changed dataset is detectable.
- **hardware model** — what it ran on (the timing numbers mean nothing without it).
- **timestamp** — when.
- **random seed** — the seed(s), tying back to the recorded streams above.

"Minimal-sufficient" is the standard: every field is there because *removing* it breaks reconstruction (drop the commit hash and you cannot recover the code; drop the data hash and a changed input is invisible), and nothing beyond that is recorded for its own sake. The archive is **auto-recorded** — produced by the run itself, not assembled by hand afterward, because a provenance record a human has to remember to write is a provenance record that is sometimes missing.

This archive is the **machine version of the append-only 历程** (the lab notebook — the human-written, append-only record of what was run and what it showed, that [ledger](../../ledger/SKILL.md) relies on). The division of labor is exact: the machine packs the full archive under a run id; the **历程 records only the run id** and the human-meaningful note ("ran the tightness sweep, method B's advantage held to 2x slack"). Because the heavy, error-prone provenance lives in the auto-recorded archive and the 历程 holds only the pointer, **the evidence chain closes itself** — every line of the notebook resolves through its run id to a fully reconstructable run, with no human asked to transcribe a commit hash correctly by hand.

### The clean-commit gate (`clean-commit-gated`)

Formal experiments run **only from a clean commit.** An **uncommitted working tree makes the run refuse to start** — one guard line at the top of the run entry point: if `git status` is not clean, abort before producing a single number.

This is one line and it is load-bearing. Without it, the archive records *a* commit hash — the last commit — while the code that actually ran included **uncommitted edits** that the hash does not capture. The recorded commit and the executed code **silently diverge**, and the divergence is invisible until it matters most: the **reviewer's question three months later** ("can you rerun configuration X to check Y?") triggers a replay from the recorded commit — which **reconstructs the wrong state**, because the real run carried changes that were never committed. You get a number that does not match your paper, from a replay you trusted, and no way to tell which run was the real one. The clean-commit gate makes that failure structurally impossible: if the tree is dirty, there is no run, so the recorded hash *always* describes the code that ran. (This is also why the formal run is gated, not the exploratory MVP — exploration is allowed to be messy; the moment a number is destined for the paper, the gate fires.)

## The agent's signature failures, at the harden-and-provenance stage

Everything above assumes the agent is the **means** — the refactorer, the checker-writer, the archive-wirer — and never the **author** of the numbers. Its gradient points at *clean and passing*, and at this stage that produces three specific, recurring failures. Name them so you catch them.

1. **The clean-but-drifted refactor.** Handed the hardening, the agent produces beautifully restructured code whose tests pass — and which computes a subtly different number than the MVP did. It reports the migration done, because from inside its optimization "the tests pass" *is* done. It cannot see the drift: the tests encode the new behavior. The **regression-against-the-MVP** match is the only counterweight, and it is yours to run and yours to insist on; a passing test suite is not it.

2. **The captured feasibility checker.** Asked to write the checker, the agent reuses the solver's constraint logic — it is right there, it is convenient, and it makes the checker pass. But a checker that shares the solver's logic shares its **blind spot** and rubber-stamps the very infeasible solutions it exists to catch. The independence requirement — *shares no logic with any solver* — is the rule, and you verify it by reading the checker's imports, not by trusting that it "passes."

3. **The silenced determinism.** Told the GPU run is slow, the agent "helpfully" leaves the determinism switches off (they cost speed) or sets one global seed (it is simpler), and the comparison numbers become unreplayable or shift under unrelated edits. Neither is a thing it will flag — both make the run *faster* or *simpler*, which its gradient reads as better. The **recorded-not-eliminated** rule and the **on-for-key-comparisons** GPU rule are yours to impose; the speed cost is a price you decide to pay, not one the agent will choose for you.

All three share a root: the agent optimizes the *legible proxy* — tests pass, code is clean, the run is fast — over the thing that actually matters, which is that the number is correct and replayable. The whole discipline of this stage — the four investments under a frozen interface, the regression match, the recorded streams, the self-closing archive, the clean-commit gate — is the counterweight. Command it to harden and to record; verify the regression match, the checker's independence, and the determinism yourself.

## What carries forward

Out of STAGE 0 you carry experiment code hardened to the research standard — everything config-ized, one shared metric path, an independent feasibility checker running on every produced solution, the interface frozen — and **proven** to preserve the MVP's numbers by a same-config-same-seed regression match. Out of STAGE 1 you carry a run that records its own randomness on independent derived seeds, achieves task-level determinism (same config and seed, same result on rerun, reproducible across machines to the reported precision), auto-packs a minimal-sufficient archive under one run id, and **refuses to start from a dirty tree.** Together these are the *trustworthy* and the *replayable*: STAGE 2 ([pipeline-and-integrity.md](pipeline-and-integrity.md)) can now run this code idempotently — because task-level determinism makes a completed cell genuinely done — and STAGE 4 can enforce anti-mixing — because every number ties to a version-consistent, fully provenanced run. The correctness you built here and the records you stamped are what let every later stage trust that a number means what its run id says it means.
