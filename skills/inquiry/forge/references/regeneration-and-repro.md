# Regeneration & Reproduction — every number rebuilds from one command (STAGE 5)

This reference is the depth behind **STAGE 5 — Regen** of the [../SKILL.md](../SKILL.md) flight plan. By the time you reach it the run is over in the only sense that matters to the cluster — the jobs have finished, the results are version-tagged, and they live in a single store. STAGE 5 is what turns that store into an *evidence base*: a thing the paper rebuilds from by one command, a thing that reproduces in a room that is not yours, and a thing whose every formal number traces back to the run that produced it. This file backs the two gated checks of the stage — `one-command-regeneration` and `repro-package-smoke-tested` — and states the four exit conditions that mean the whole step is forged, not merely finished.

The governing fact, restated because every move below is judged against it:

> Every table and figure the paper will contain is generated by a SCRIPT from the results store, by ONE command, with NO manual number-editing anywhere in the chain. A hand-edited spreadsheet is the number-one channel for untraceable error.

---

## One-command regeneration is the 纲 of the whole step

The step has six stages, but it has one governing standard, and this is it. *纲* — the main rope of a net, the thing that pulls every other strand taut when you lift it. One-command regeneration is that rope here, and the reason is that a single bar buys three things at once that you would otherwise have to chase separately:

- **Reproducibility.** A number that rebuilds from the raw results by a command anyone can run is a number anyone can reproduce. The script *is* the proof of how the figure was made — there is no gap between "the result" and "how the result was computed" for an error to hide in.
- **Anti-rot.** Results rot silently. A figure hand-copied into the paper in March drifts from the store it came from the moment a run is re-done in April, and no one sees the drift because the eye cannot diff a column of numbers against a spreadsheet cell pasted three weeks ago. When the figure is *generated*, re-running the command after the April rerun either reproduces the same number or visibly changes it — the drift becomes loud instead of silent.
- **Most of the reproduction package, for free.** The script that regenerates the main table is the same script the repro package needs to ship. Build it once, for your own paper, and the package is most of the way written as a side effect of doing the analysis you had to do anyway.

The hand-edited spreadsheet is the single channel this standard closes. It is where a transposition, a stale paste, a wrong-column copy, a forgotten unit conversion enters the paper untraceably — *untraceably* because once a number is typed into a cell, nothing connects it back to the run that produced it, and no checker can find the error because there is no chain to check. Forbidding the manual edit anywhere in the chain is not fussiness; it is removing the one place where an error has no provenance.

There is a second payoff, and it is what makes the *next* step (reckoning — results analysis) cheap. When every figure is a script reading the store, every analysis question becomes a one-line edit and a rerun. Changing a figure's definition (plot the median, not the mean), adding seeds (the store grew; rerun the same command and the error bars tighten), swapping a statistical test (a paired test where a Welch's was) — each is *edit the script, rerun one command*, not a manual re-derivation of every affected cell. The discipline you pay for here is repaid every time you ask a new question of the same data. A run whose figures are hand-assembled makes the analysis loop expensive precisely when you most want it to be cheap — when a reviewer asks for one more cut of the data the week before the deadline.

The agent's role here is exactly the labor this standard creates: it writes the generation scripts (read the store, emit the table or figure), wires them behind one command, and re-runs that command on demand. It does *not* author the numbers — the numbers come from the store, and the store is frozen output of the operator stage. A regeneration script that quietly recomputes a metric, filters a row, or "cleans" a value is the operator boundary breached one stage late; the script's only job is to *render* what is on disk, never to change it.

```
Write a regeneration script for each table and figure the paper will contain.
Each script READS from the single results store (point to it) and EMITS the
table (LaTeX/CSV) or figure (the plot file) — nothing else.

Rules:
  - NO number is typed by hand anywhere. Every value in the output traces to a
    query against the store.
  - The script does NOT recompute metrics, filter rows, or "clean" values — the
    store is frozen; render it, do not change it. If a number looks wrong, FLAG
    it to me; do not fix it in the script.
  - Wire all of these behind ONE command (a Makefile target / a single entry
    script) so `make tables` (or equivalent) rebuilds EVERY table and figure.
  - The command must be re-runnable: running it twice produces byte-identical
    output (or output that differs only where the store genuinely changed).

Output the one command and the list of artifacts it regenerates.
```

## The reproduction package — accumulate as you go, never cram before submission

The reproduction package is the thing a reader needs to rebuild your results from nothing. Maintain it throughout the run as a `repro/` directory, and add to it as each experiment class completes — not in a panic the week of submission. A package assembled at the end, from memory, against a deadline, is the package that ships broken; a package grown alongside the work is correct because each piece was added when the experiment that needs it was fresh.

The `repro/` directory holds four things:

| Component | What it is | Why it is in the package |
|---|---|---|
| **environment lockfile** | the exact pinned dependency set (a lockfile, or a container image tag) | "it worked" is meaningless without the environment it worked in; the lockfile is what makes the clean-room rebuild reconstruct *your* environment, not a newer one that broke |
| **data fetch-and-preprocess scripts** | the scripts that pull the raw data and transform it into the form the experiments consume | a reader who cannot regenerate your inputs cannot regenerate your outputs; the preprocessing is part of the result, not a precondition assumed away |
| **one-command main-table script** | the regeneration command from the section above, packaged to run from the repro directory | this is the payoff of one-command regeneration crossing into the package — the same script that built your paper's tables builds the reader's |
| **README** | the ordered, literal steps from a clean checkout to the regenerated main tables | the entry point; the thing the smoke test executes; the difference between a package and a pile of scripts |

Each **experiment class adds its command as it completes.** When the ablation finishes, its regeneration command goes into the package and the README; when the sensitivity scan finishes, its command goes in; and so on. The package is never a separate task with its own deadline — it is the running tail of the work you have already done, and at submission it is already complete because you never let it fall behind.

```
Maintain a repro/ directory throughout this run. It holds exactly:
  1. the environment lockfile (or container image tag),
  2. the data fetch-and-preprocess scripts,
  3. the one-command script that regenerates the main tables,
  4. a README with the LITERAL ordered steps from a clean checkout to the
     regenerated tables.

As EACH experiment class completes (main table, ablation, sensitivity, ...),
append its regeneration command to the package and to the README — do not defer
this to submission time. Tell me each time you add one, and what it regenerates.
```

## The clean-environment smoke test — the agent's job, on purpose

A reproduction package's most common cause of death is **"works on the author's machine."** The package depends, invisibly, on something the author's environment has and a stranger's does not — a globally installed tool, an environment variable set years ago in a shell profile, a data file sitting in a home directory the scripts assume, a dependency that the lockfile failed to pin because it was already present. None of this shows up when *you* run the package, because your machine is the one place every hidden dependency is satisfied. It shows up the instant someone else runs it — which, for a reproduction package, is the moment that matters.

The defense is to run the package yourself, periodically, **from scratch, in a fresh clean environment** — a new container, a fresh checkout, nothing preinstalled — executing the README literally, step by step, exactly as written, and fixing the package wherever the README and reality diverge. This is the smoke test, and it is exactly the kind of job the agent is best at and you should hand it: it is dull, it is mechanically well-bounded ("do what the README says, in order, in a clean box, and report the first line that fails"), it has an unambiguous pass condition (the main tables regenerate), and it is tedious enough that a human does it carelessly and an agent does it the same way every time. The agent runs the README from a clean checkout, and every "command not found" or "no such file" it hits is a hidden author-machine dependency caught before a reviewer catches it.

The pass condition is precise: from a clean environment, following the README's steps in order, the main tables **regenerate** — the same numbers the paper reports, by the package's own one command, on a machine that knew nothing about your project an hour ago. Anything less than that and the package does not reproduce; it only reproduces *for you*, which is to say it does not reproduce.

```
Run the reproduction smoke test. In a FRESH, CLEAN environment (new container or
equivalent — nothing preinstalled, fresh checkout):
  1. Follow repro/README literally, step by step, in order. Do NOT use any tool,
     file, env var, or data that the README does not explicitly install/fetch.
  2. At the FIRST step that fails, STOP and report: the step, the exact error,
     and the hidden dependency it reveals (a global tool, an env var, a data file
     assumed present, an unpinned dependency).
  3. The PASS condition is: the main tables regenerate by the package's one
     command, matching the paper's numbers. Nothing weaker counts.

Report pass/fail and the full list of fixes the package needs. Re-run after each
fix until a clean environment regenerates the tables from the README alone.
```

## The exit conditions — where the run actually ends

The run is not over when the cluster jobs finish. "The jobs all completed" is a statement about compute, not about evidence; the evidence base is forged only when four conditions hold at once. State them as the terminus:

1. **Every protocol number is on disk and version-consistent.** Every number the frozen protocol called for exists in the single results store, and every row carries the code version that produced it, with no table drawing from more than one version (the *混表* — mixing old and new numbers in one table — defense from the integrity stage holds). There is no number the paper needs that is still only on a screen, in a log, or in someone's memory.
2. **Any table or figure regenerates by one command.** The governing standard is met in full: pick any table, any figure, run the one command, and it rebuilds from the store with no hand edit in the chain.
3. **The clean-environment reproduction smoke test passes.** The package, run from a fresh checkout in a clean box following the README literally, regenerates the main tables. "Works on my machine" has been killed on a machine that is not yours.
4. **The 历程 evidence chain is complete.** The *历程* (the append-only lab notebook) closes the loop: every formal run is traceable to its **run id**, and through the run id to its commit, its config, and its seed. A reviewer's question three months out — "where did this number come from?" — has a bit-for-bit answer, because the 历程 records the run id and the run id reconstructs the exact state that produced the number.

When those four hold, stop running. The thing you carry out is a *regenerable, reproducible, traceable evidence base* — not a finished job queue.

It is worth naming what the human role has shrunk to by this point, because it is the proof that the run was operated correctly. Done right, everything in STAGE 5 is the agent turning: it writes the regeneration scripts, grows the repro package, and runs the clean-room smoke test, all of it dull, bounded, mechanical labor it does well and tirelessly. Two events, and only two, remain yours across the whole run, and both are decisions no operator can make:

- **Protocol-change decisions** — whether a discovered bug forces a full rerun, whether to add seeds, whether the frozen interface must be cut to a new version. These reshape what counts as evidence, and they are bets, not operations.
- **Anomalies the agent cannot auto-attribute** — the failure that is neither a clean resource error nor a clean code error, the number that is suspicious but feasible. Whether such an anomaly is a bug or a result is a judgment, and the agent's gradient (please you, report the green check) points the wrong way on exactly these.

Everything else is the agent's. If you find yourself doing more than those two things in this stage, something earlier slipped — a number that should be in the store is being assembled by hand, a package that should have grown as you went is being crammed, a check the machine should run is being run by your eye. The shrink to two events is not a convenience; it is the measure of whether the run was forged or merely finished.

## What the two gated checks demand

`one-command-regeneration` is clear when every table and figure the paper will contain is produced by a script reading the single results store, wired behind one command, with no hand-edited number anywhere in the chain — and the command is re-runnable to byte-identical output where the store is unchanged.

`repro-package-smoke-tested` is clear when the `repro/` directory holds the four components (lockfile, data scripts, one-command main-table script, README), was accumulated as the experiment classes completed rather than crammed at the end, and the agent has run the README from scratch in a fresh clean environment and the main tables regenerated. The step as a whole is done only when all four exit conditions above hold together — the two checks plus version-consistency on disk and a complete 历程 chain.
