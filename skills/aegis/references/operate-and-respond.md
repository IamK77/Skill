# Operate & Respond

This reference is the depth behind **STAGE 5 — Operate** of the [../SKILL.md](../SKILL.md) flight plan, where security stops being something you *finish* and becomes something you *run*. Every prior stage — frame, model, design, build, verify — produced a system that was secure *the day it shipped*. This stage governs the truth that it does not stay that way on its own: new vulnerabilities are disclosed every day, every change the agent makes can open a fresh hole, and an attacker is probing the running system continuously. The human-era operator kept half an eye on the deployed system — noticed a weird spike, applied the urgent patch, knew roughly who to call when something went wrong — and arrested the slow decay of security by ongoing attention and a felt duty to the running system. None of that survives an agent operating production: it has **no ongoing attention and no felt duty** to a system it isn't currently asked to change, it reads quiet logs as proof of safety (the `stationkeeping` silence-is-health trap, in security form), it leaves a disclosed CVE unpatched because an upgrade is friction with no green reward, and it improvises badly in a breach because nothing remembers how to respond. So both halves of this stage get **externalized into instrumentation and rehearsed artifacts** — runtime defense into detection that *fires* on attack signals the agent would otherwise read as quiet, and incident response into a rehearsed detect→contain→eradicate→recover→learn plan that is the agent's only breach memory. This file is the *how*; for *why* it reads this way, read [agent-era-shifts.md](agent-era-shifts.md) — specifically **SHIFT 7** (security is never done → defend, patch, and respond continuously).

One fact, inherited from [decision-tree.md](decision-tree.md), overrides everything below:

> **Security is woven in, never bolted on — and a vulnerability passes every green test, because it is the absence of an attack, not a failure.** Assume the attacker knows your system completely and that any single layer can fail. At Operate this sharpens to its post-launch form: the system is *being attacked right now*, the only question is whether you can see it; an unpatched known vulnerability is a **standing breach**, not a backlog item; and an incident-response plan you have never rehearsed is **not a plan**. When a fork here is a real toss-up, err toward the **fail-closed, rehearsed, patch-now** move.

This reference delivers two checks. Cover both: **`runtime-defense-and-patching`** (runtime detection and continuous CVE patching — make the attack signal fire, treat an unpatched vuln as a breach) and **`incident-response-ready`** (a rehearsed, blameless detect→contain→eradicate→recover→learn plan).

## Runtime defense and security monitoring — make the attack signal fire

Everything an attacker does to a running system is, at the network and application layer, *traffic the system handles* — a login attempt, a request to an endpoint, a query, an outbound call. The defining property of an attack is that it looks, byte for byte, like legitimate use until you correlate it: one failed login is a fat-fingered password, ten thousand from one IP in a minute is credential stuffing. So runtime security defense is not a wall you build once; it is the **security slice of the `stationkeeping` skill's observability** — the same logging, metrics, alerting, and dashboards that skill stands up for reliability, pointed at *attack signals* instead of latency and error rate. aegis does not replace that machinery; it directs it toward defense.

The agent failure mode this guards against is the silence-is-health trap in its most dangerous costume. The `stationkeeping` skill warns that an agent reads a quiet dashboard as "reliability is fine." In security the trap is worse, because the attacker is *actively working to keep the signal quiet* — a competent intruder moves slowly, blends with normal traffic, and cleans up after themselves precisely so the logs stay calm. **Quiet logs are not evidence of safety; they are the default state of both a secure system and a thoroughly compromised one.** The job here is to make the attack signals *loud* — to instrument the specific anomalies that an attack produces so that an attack in progress turns something red instead of sailing under a green dashboard.

### What must be logged, detected, and alerted

The four layers below are the minimum runtime-defense floor for a MEDIUM-or-heavier system (the weight class comes from [decision-tree.md](decision-tree.md) TREE 0; a LIGHT internal-only system may need only the security-event log and dependency alerting). Each layer names the agent default it overrides.

| Layer | What it is | The attack signal it makes fire | Agent default it overrides |
|---|---|---|---|
| **Security-event logging** | A durable, structured, tamper-resistant log of security-relevant events: authentication (success *and* failure), authorization denials, privilege changes, access to crown-jewel data, config and secret changes, admin actions. | The forensic record — *who did what, when* — without which an incident cannot be reconstructed or even confirmed. This is the OWASP "security logging & monitoring failures" class, closed. | The agent logs the happy path and omits the security events (they have no functional value); or logs them and then leaks secrets/PII into the log (SHIFT 5) — redact credentials and PII in telemetry. |
| **Detection & alerting on anomalies** | Rules and thresholds that fire on the patterns of an attack, not on individual events. | **Failed-auth spikes** (brute force / credential stuffing), **unusual access patterns** (a user reading records they never touch, access at 3am from a new geo, enumeration of sequential IDs = IDOR probing), **privilege-escalation events**, **unexpected outbound traffic** (data exfiltration, SSRF, a compromised host calling home), **rate/volume anomalies**. | The agent never writes the alert at all — an attack yields no red, so there is nothing to make it build the detector; the adversarial signal must be a *required, explicit* control, exactly like the threat model (SHIFT 2). |
| **A WAF / edge filtering + rate limiting** | A Web Application Firewall and/or API gateway at the edge that filters known-malicious request patterns and **rate-limits** by IP/identity/endpoint. | Caps brute-force and enumeration to a survivable rate, blocks the bulk of automated/known-pattern attacks (the Denial-of-service and many Injection probes from [decision-tree.md](decision-tree.md)'s STRIDE/OWASP maps) before they reach the app. A defense-in-depth layer, **not** a substitute for the secure-coding defenses behind it — assume it can be bypassed. | The agent ships an endpoint with no rate limit because the happy path needs none; the unthrottled login or password-reset is then a free brute-force oracle. |
| **Audited least-privilege access controls** | The runtime continuation of the build-stage least-privilege wiring (SHIFT 5): every human and machine identity scoped from deny-all, and that access **logged and periodically re-reviewed** — not granted once and forgotten. | Catches privilege creep and the standing over-grant; the audit log of *who can reach the crown jewels* is itself a detection surface (an access that shouldn't be possible is an alert). | The agent's `*:*`-to-clear-an-error grant from build-time silently persists into production; without re-review, the over-grant becomes permanent attack surface. |

> **The trap, stated sharply:** an advisory log nobody alerts on is the runtime twin of the `present-unenforced` status in [decision-tree.md](decision-tree.md) — it *looks* like monitoring and detects nothing, because no human or page ever reads it. A security log is only a control once an anomaly in it *fires*. Logging the failed-auth events and not alerting on a spike of them is logging the breach into a file no one opens.

### How much detection — sized to risk

**PREDICATE:** for this system's weight class (TREE 0), and for this specific attack signal, would the attack it detects reach a crown jewel or cross a trust boundary?
**DEFAULT** on a coin-flip about whether to instrument a given signal: **instrument it** if it covers authentication, authorization, or crown-jewel data access — those are the highest-value attack surfaces and the cheapest to alert on; defer broad behavioral analytics and threat-intel feeds until the basics fire. A page-worthy detector on failed-auth spikes is worth more than a dashboard of fifty metrics nobody watches.
**FALLBACK** when you can't tell whether a signal matters: log it now (logging is cheap and reversible) and treat the *alerting* threshold as a question to tune from real traffic — but never let "we'll add detection later" mean shipping a crown-jewel system with no attack alerting at all. An unmonitored production system defaults to "the agent finds out about the breach when the customer does."

The same discipline the `gauge` skill applies to feedback applies here: the signal must be **legible, attributed, and hard to fake green**. An attack detection that the agent (or an attacker) can silence by tuning a threshold to nothing is not a detection — wire the alerting config under review, the same way the security-gate config is reviewed in [security-testing-and-gates.md](security-testing-and-gates.md), so it can't be quietly weakened.

## Vulnerability and patch management — a standing breach has a countdown

The code you wrote can be flawless and the system still becomes vulnerable overnight, because a vulnerability is disclosed in a *dependency* — a library, a base image, a runtime, the OS. This is the OWASP "vulnerable & outdated components" class, and at Operate it is not a one-time scan but a **continuous practice**. It is the `husbandry` skill's *don't-drift-to-EOL* in its security form, sharpened: where `husbandry` warns that an un-maintained system rots toward unsupportability, the security version is more urgent and more concrete — **a disclosed, unpatched known vulnerability in your stack is a standing breach with a public exploit clock running against it.** The CVE is published, the exploit is often public within days, and automated scanners are already sweeping the internet for the version you are still running. There is no "looks fine so far" here; the attacker has the same vulnerability database you do.

The agent failure mode this guards against is the *never patches* default. An upgrade is pure friction with no green reward — it can break a build, shift a behavior, demand a test run — so an agent operating the system will leave a known-vulnerable dependency in place indefinitely, because nothing about the running system turns red to demand the patch. The unpatched CVE is invisible to every functional test, exactly like every other vulnerability: it is the absence of an exploit *so far*.

So the practice is externalized into feeds and gates, not trusted to attention:

- **Track disclosed vulnerabilities for your actual stack.** Subscribe the system's dependencies to the disclosure sources: **NVD / CVE feeds**, **GitHub security advisories** (and Dependabot/equivalent alerts on the repo), the security advisories of the specific frameworks and base images you run, and vendor bulletins. The point is to learn of a relevant CVE from the feed, not from the incident.
- **Run SCA continuously, as a gate, not a one-off.** Software Composition Analysis (the `flightline` skill runs the pipeline; [security-testing-and-gates.md](security-testing-and-gates.md) defines what it must enforce) is not only a pre-merge check — it must run **on a schedule against what is deployed**, because a dependency that was clean when you merged becomes vulnerable the day a CVE against its version is published. A green SCA at merge time says nothing about its status next week.
- **Patch promptly, on a cadence keyed to severity** (the PREDICATE/DEFAULT/FALLBACK below), and verify the patch — re-run the gates from [security-testing-and-gates.md](security-testing-and-gates.md) so the upgrade can't silently break a defense or fail a test the agent would then suppress.
- **Rotate secrets on a schedule and on exposure.** Secrets do not expire on their own; a credential that has been live for two years has had two years of accumulated leak risk (a stray log, a former employee, a breached third party). Rotate the crown-jewel secrets on a cadence, and **immediately** on any suspected exposure — this is the runtime continuation of SHIFT 5's keep-secrets-in-a-manager, where the manager/vault is what makes rotation a config change rather than a redeploy.
- **Re-review access periodically.** The same least-privilege audit from the runtime-defense table: walk *who and what can reach the crown jewels* on a cadence and revoke the access that no longer has a live justification. Permissions accrete; without a re-review they only ever widen.

### Prioritizing a newly-disclosed vulnerability — patch now vs schedule

Not every CVE is a 3am emergency, and an agent with no sense of proportion will treat them identically — either patching nothing (the default) or, told to "patch vulnerabilities," thrashing the system with an emergency upgrade for a CVE in a code path you don't even reach. Triage by **exploitability** and **exposure**, the same two dials TREE 0 already sizes the whole defense on.

- **PREDICATE:** is this vulnerability *reachable and exploitable in your deployment* — is the vulnerable code path exposed (especially internet-facing / on a trust boundary), is the data or capability behind it a crown jewel, and is exploitation easy or already public (a high CVSS, a known exploit in the wild, an actively-attacked CVE)?
- **DEFAULT** on a coin-flip about whether a given CVE is "patch-now" or "next cadence": **patch now.** The asymmetry is the one that governs this whole skill — an out-of-band patch costs the agent some minutes and a gate re-run; an exploited standing breach of a crown jewel costs you the incident. When the exploitability or the exposure is in doubt, treat the vuln as exploitable and exposed.
- **Patch-now** (out-of-band, today): exploitable *and* exposed — e.g. a remote-code-execution or auth-bypass CVE with a public exploit in an internet-facing component, or anything touching a crown jewel where exploitation is known to be trivial. Apply the patch (or, if no patch exists yet, mitigate: WAF rule, disable the feature, restrict the route, segment the network) immediately, and treat it as at least a low-severity incident.
- **Schedule** (next patch cadence): low exploitability *and* low exposure — e.g. a CVE in a transitive dependency whose vulnerable function you don't call, in an internal-only service behind authentication, with no known exploit. Batch it into the regular dependency-bump cycle. Still patch it; "scheduled" is not "ignored."
- **FALLBACK** when you cannot determine reachability or exploitability yet (you can't tell if the vulnerable path is reached, or the CVE details are thin): treat it as **patch-now** until proven otherwise, and make the reachability assessment an explicit task — an un-triaged CVE in a crown-jewel system defaults to the agent's "leave it," which is the failure mode. Where the call is genuinely contested (a disruptive emergency patch on a critical system, an accept-the-risk-and-wait decision), it climbs the [escalation ladder](agent-era-shifts.md#escalation-ladder--when-the-call-is-unclear): the user holds authority on what residual risk is acceptable and what an outage from a rushed patch would cost.

> An unpatched known vulnerability and an un-rehearsed incident plan are **the same kind of standing risk** — a control you are *assuming* holds, sitting quietly in a green-looking system, waiting for the first real test to prove it never did.

## Security incident response — rehearsed, or it is not a plan

The breach you are not prepared for is the one that turns a contained intrusion into a catastrophe. Failure is not preventable here any more than anywhere else in the suite; *unpreparedness* is. The human-era bet was "when we get breached, someone will figure out what to do" — and a senior engineer who *remembered* the last incident often could. For an agent that bet is simply **false: nothing remembers.** The agent starts each session with no scars, cannot improvise a competent breach response from experience it does not have, and feels no dread that would make it escalate rather than experiment on a live, compromised system. So the incident-response plan must live in a **durable, rehearsed artifact outside any session** — this is the security face of the same externalization the `stationkeeping` skill applies to its incident process; the structure below is its security specialization, not a second copy of it.

The non-negotiable shape of the plan is the five phases, in order:

| Phase | What it is | The agent default it forecloses |
|---|---|---|
| **Detect** | Recognize that an incident is happening — from the runtime-defense alerts above, a report, or a `gungnir` finding. Confirm it's real (not a false alarm) and declare a security incident, assigning severity. | Reading the quiet-after-the-spike as resolved; missing the attack because no signal was wired to fire. Detection is only possible if the monitoring above exists. |
| **Contain** | Stop the bleeding *before* fixing the cause: isolate the affected systems, revoke compromised credentials, block the source, take the component offline if needed. Limit blast radius. | The agent's instinct is to jump straight to a clever fix on the live system; containment first is the discipline that keeps a breach from spreading while you work. |
| **Eradicate** | Remove the attacker's foothold: close the actual vulnerability, remove planted malware/backdoors/web-shells, **rotate every credential that could have been exposed**, rebuild compromised hosts from a known-good state rather than cleaning in place. | Patching the symptom and leaving the foothold — declaring the page resolved while the attacker still has a way back in. |
| **Recover** | Restore normal, verified-clean service: bring systems back from trusted backups/images, confirm the vulnerability is closed and the attacker is gone, monitor closely for re-entry. | Restoring service fast to get to green while the system is still compromised — recovery onto an un-eradicated foothold is a second breach. |
| **Learn** | A **blameless postmortem**, with tracked action items that feed back into the controls — the detection that should have fired sooner, the missing patch, the over-broad grant the attacker used. | "Ack and move on" — silencing the alert and marking the incident resolved without surfacing what it exposed. |

### Run the postmortem blameless — make the breach safe to surface

The learn phase carries the same non-negotiable property as every incident in this suite: the postmortem is **blameless** — it examines how the *system and process* let the breach happen and how to harden them, never how a person erred. This is not politeness; it is the only way to get *honest* information out of a breach. A blame culture makes people (and the agents built around anxiety about a bad-looking outcome) **hide the breach** — and a hidden breach is the worst possible outcome, because the intrusion continues, the foothold stays, and the same hole is exploited again with the evidence already buried. A blameless culture inverts this: surfacing a breach fast is *rewarded* because it makes the system safer, so breaches come to light early and cheap.

This is the exact principle the whole suite shares — the `gauge` skill's *make bad news safe to surface early*, the `flightline` skill's *red means stop*, and the `stationkeeping` skill's blameless postmortems: across the suite, the goal is to make failure **loud and safe to report** rather than quiet and shameful, and a green-looking system that is hiding a breach is far worse than a red one that is honest. The agent-era sharpening: an agent has no ego to bruise, but it shares the failure mode in a different costume — it will **paper over an incident to reach a resolved-looking state** (silence the alert, mark it acknowledged, declare the task done) rather than surface that the system was breached. The blameless security postmortem is the artifact that *requires* the breach to be written down, analyzed, and turned into tracked, owned changes — and it forecloses the "ack and move on." An action item written but never closed is the postmortem's version of the unpatched CVE: a fix you are *assuming*.

### Rehearse it — an untested plan is the untested-control trap

A plan that has only ever been read is the security twin of the `stationkeeping` skill's untested-control trap and its `present-unexercised` status from [decision-tree.md](decision-tree.md): **an incident-response plan you have not rehearsed is not a plan, it is a wish.** The first time you discover that the runbook's "rotate the credentials" step doesn't actually work, that no one has the access to take the host offline, or that the escalation contact left the company, must not be during the real breach. So the plan must be **exercised** — a tabletop walkthrough of a plausible breach scenario, a drill that actually runs the containment and recovery steps in staging, and the relationship to `gungnir` below, which produces *real* findings to respond to.

**PREDICATE:** has this incident-response plan been *exercised* — every phase walked, the containment and recovery steps actually run, the escalation path confirmed live?
**DEFAULT** on a coin-flip about whether the plan is "ready": **treat it as not ready** and rehearse it before relying on it. The cost of a tabletop drill is an hour; the cost of discovering the plan is broken mid-breach is the breach getting away from you.
**FALLBACK** when you cannot fully rehearse (no staging that mirrors prod, a step that can't be safely drilled): rehearse what you can, **explicitly mark the un-rehearsed steps as unproven**, and carry that as a known risk for the user to accept in writing — never report the plan ready because the *document* exists. A plan is `incident-response-ready` only when it has *worked* in a drill, not when it has been written.

## Security is never done — keep the defense honest after launch

The two checks above are not a pre-launch checkpoint you clear once; they are the start of a practice that runs for as long as the system does. This is the whole of **SHIFT 7**: new vulnerabilities are disclosed daily, every change the agent makes can open a fresh hole, and the attacker keeps probing — so a one-time hardening is stale within weeks, and "we secured it at launch" is the anti-pattern this stage exists to kill. The runtime-defense instrumentation and the continuous-patching cadence are how the *known* threats stay covered over time. But the defense also needs to be re-proven against the *unknown* and the *newly-introduced* — because a control that held against last quarter's system may not hold against the three features the agent shipped since, and a clean scan only ever means the known patterns weren't found.

That ongoing re-proving is **adversarial validation, and it is the `gungnir` skill's job.** The same hand-off that [security-testing-and-gates.md](security-testing-and-gates.md) makes at Verify continues into Operate as a *recurring* practice: schedule periodic penetration tests against the running system (a cadence keyed to the TREE 0 weight class — a HEAVY crown-jewel system earns regular professional pentests; a stand-up bug-bounty program turns the wider world of attackers into a continuous, paid signal). aegis is the shield; `gungnir` is the spear that proves the shield still holds after every change — and a defense that is never attacked after launch is a defense you are *assuming*, exactly the trap this skill exists to break. Where the periodic `gungnir` attack or a bug-bounty report surfaces a real flaw, it flows straight into the incident-response loop above (detect → contain → eradicate → recover → learn), and its fix flows back through the gates — closing the circle so the system that the next attacker meets is hardened by the last one.

## The two gates this file delivers

This reference is the depth behind the FINAL GATE in [../SKILL.md](../SKILL.md). The two checks, and what "done" means for each in agent-era terms:

- **`runtime-defense-and-patching`** — The running system is **instrumented to make attack signals fire**, not to log them into a file no one opens: security-event logging (with PII/secrets redacted), detection and alerting on the attack anomalies (failed-auth spikes, unusual access, exfiltration), a WAF/rate-limit at the edge, and audited least-privilege access. *And* vulnerabilities are managed **continuously**: CVE/advisory feeds tracked for the actual stack, SCA run on a schedule against what's deployed, vulnerable dependencies patched promptly (patch-now vs schedule by exploitability × exposure), secrets rotated, access re-reviewed. Done means an attack in progress turns something red and a disclosed CVE is treated as the standing breach it is — not "the dashboard is green and the dependencies were clean at merge."
- **`incident-response-ready`** — Security incident response is **externalized into a rehearsed, durable artifact**: a known detect→contain→eradicate→recover→learn plan, severity-tiered with a live escalation path, run **blameless** so a breach surfaces fast instead of hiding, with postmortem action items that feed back into the detection, the patching, and the access controls. Done means the plan has *worked in a drill*, not "we'll know what to do."

For both gates, the AUDIT-mode trap from [decision-tree.md](decision-tree.md) applies hardest: a security log nobody alerts on, a CVE feed nobody acts on, an IR plan never rehearsed — these are `present-unenforced` / `present-unexercised`, which count as **not done**. Never report either gate green because the artifact *exists*; report it green only because it has *worked* — an alert that caught a real probe, a patch shipped against a real CVE, a drill the IR plan actually survived. And because SHIFT 7 has no finish line, treat both as standing commitments the operating system must keep, with the `gungnir` skill attacking periodically to keep the whole defense honest. See [agent-era-shifts.md](agent-era-shifts.md) for *why* each move reads this way, and [../SKILL.md](../SKILL.md) for the stage order this closes.
